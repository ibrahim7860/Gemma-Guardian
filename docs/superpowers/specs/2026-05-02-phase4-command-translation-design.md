# Phase 4 — Operator Command Translation Path + Tickers + Allowlist Hardening

**Status:** DRAFT — awaiting approval

## 1. Goals

Phase 4 closes the second outbound interaction path on the dashboard — the multilingual command box that is the headline demo moment for "Gemma 4 speaks N languages, no cloud translation." It also lands two pieces of polish surfaced during Phase 3 review:

1. **Operator command translation round-trip** — Flutter sends `operator_command` → bridge → Redis → EGS Gemma 4 E4B → Redis → bridge → Flutter `command_translation` → operator clicks DISPATCH → `operator_command_dispatch` propagates → operator sees confirmation. Stub EGS for local dev so this path is independently testable today (Person 3's real EGS lands later and plugs in unchanged).
2. **`finding_id` allowlist on bridge** — reject inbound `finding_approval` frames whose `finding_id` is not in the aggregator's known set. Closes the Phase 3 adversarial-review finding.
3. **Validation event ticker on drone status panel** — render `state_update.recent_validation_events` (currently arriving but ignored) so judges can see Gemma 4 self-correcting in real time.

## 2. Non-goals

- **Real EGS Gemma 4 E4B integration.** Person 3's seat. Phase 4 ships a `scripts/dev_command_translator.py` stub that consumes from `egs.operator_commands` and publishes deterministic translations to `egs.command_translations`. Person 3 swaps the real EGS in without bridge or frontend changes.
- **`set_priority` and `set_language` execution semantics.** The bridge republishes structured commands; only `recall_drone`, `restrict_zone`, `exclude_zone` need to actually do anything in the demo (and that "doing" is Person 3's). Phase 4 covers the whole command surface for translation/preview/dispatch but does not act on `set_priority` or `set_language` server-side.
- **Voice / audio operator commands.** Stretch from `docs/07`. Out of scope.
- **EGS state echo for confirmed dispatch.** Same forward-compat play as Phase 3 findings: Flutter shows a "dispatched" state on bridge ack; if Person 3's EGS later echoes a confirmation in the `egs.state` envelope, that becomes the green-check moment.

## 3. Architecture

### 3.1 Channel additions

Two new Redis channels, both JSON-validated:

| Channel | Direction | Schema | Owner |
|---|---|---|---|
| `egs.operator_commands` | Bridge → EGS | `operator_commands_envelope` (new) | Bridge publishes; EGS subscribes |
| `egs.command_translations` | EGS → Bridge | `command_translations_envelope` (new) | EGS publishes; Bridge subscribes |

`egs.operator_actions` (locked Phase 3) gains a new `kind`: `operator_command_dispatch`. Discriminated `oneOf` extends without breaking existing consumers.

### 3.2 End-to-end message flow

```
Operator types in Spanish
  │
  ▼
Flutter CommandPanel
  │ {type: operator_command, command_id, language, raw_text, contract_version}
  ▼
WS bridge — validates against websocket_messages.operator_command
  │ {kind: operator_command, command_id, language, raw_text,
  │  bridge_received_at_iso_ms, contract_version}
  ▼
Redis: egs.operator_commands
  │
  ▼
EGS (Phase 4: dev_command_translator.py stub; later: real Gemma 4 E4B)
  │ Calls Gemma 4 E4B with operator_commands schema
  │ Validates structurally + semantically (recall_drone target alive, etc.)
  │
  ▼
Redis: egs.command_translations
  │ {kind: command_translation, command_id, structured, valid,
  │  preview_text, preview_text_in_operator_language,
  │  egs_published_at_iso_ms, contract_version}
  ▼
WS bridge — validates against operator_actions.command_translation
  │ {type: command_translation, command_id, structured, valid,
  │  preview_text, preview_text_in_operator_language, contract_version}
  ▼
Flutter CommandPanel — shows preview, enables DISPATCH/REPHRASE
  │
  │ Operator clicks DISPATCH:
  │ {type: operator_command_dispatch, command_id, contract_version}
  ▼
WS bridge — validates, republishes
  │ {kind: operator_command_dispatch, command_id,
  │  bridge_received_at_iso_ms, contract_version}
  ▼
Redis: egs.operator_actions  (same channel as findings approvals)
  │
  ▼
EGS (Phase 4+ Person 3 work) — executes the command
```

The `command_id` is the correlation key end-to-end; it is generated by Flutter using the same `${sessionId4}-${ms}-${counter}` format as Phase 3 finding approvals. The same charset/length validation guards both flows because they share `_common.json#/$defs/command_id`.

### 3.3 Why two channels, not one

Could collapse into a single `egs.commands` channel with discriminated `oneOf` like `egs.operator_actions`. We don't, because:

- Direction asymmetry: `egs.operator_commands` is bridge → EGS only; `egs.command_translations` is EGS → bridge only. A single bidirectional channel invites a subscriber loop bug (bridge re-receives its own publish).
- Different validation tracks: outbound is a simple wrapper around the WS frame; inbound is a Gemma 4 output that needs the operator_commands schema.
- Person 3 can implement the EGS as two cleanly-separated subscribers/publishers without touching state shared with the dispatch channel.

## 4. Schema changes

### 4.1 `shared/schemas/operator_actions.json` — add `operator_command_dispatch`

Extend the existing `oneOf` with a second branch:

```json
{
  "operator_command_dispatch": {
    "type": "object",
    "required": ["kind", "command_id", "bridge_received_at_iso_ms", "contract_version"],
    "additionalProperties": false,
    "properties": {
      "kind": {"const": "operator_command_dispatch"},
      "command_id": {"$ref": "_common.json#/$defs/command_id"},
      "bridge_received_at_iso_ms": {"$ref": "_common.json#/$defs/iso_timestamp_utc_ms"},
      "contract_version": {"type": "string", "pattern": "^\\d+\\.\\d+\\.\\d+$"}
    }
  }
}
```

Also extract the existing inline `command_id` constraint from Phase 3 into `_common.json#/$defs/command_id` so both `finding_approval` and `operator_command_dispatch` share it. **Decision baked in:** the existing `websocket_messages.json` constraint is replaced by the same `$ref`.

### 4.2 New schema: `shared/schemas/operator_commands_envelope.json`

The bridge → EGS message:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "...operator_commands_envelope.json",
  "title": "Bridge → EGS operator_command envelope",
  "type": "object",
  "required": ["kind", "command_id", "language", "raw_text", "bridge_received_at_iso_ms", "contract_version"],
  "additionalProperties": false,
  "properties": {
    "kind": {"const": "operator_command"},
    "command_id": {"$ref": "_common.json#/$defs/command_id"},
    "language": {"$ref": "_common.json#/$defs/iso_lang_code"},
    "raw_text": {"type": "string", "minLength": 1, "maxLength": 4096},
    "bridge_received_at_iso_ms": {"$ref": "_common.json#/$defs/iso_timestamp_utc_ms"},
    "contract_version": {"type": "string", "pattern": "^\\d+\\.\\d+\\.\\d+$"}
  }
}
```

`maxLength: 4096` cap on `raw_text` prevents a 10MB DoS post on a Redis subscriber. (Same defensive pattern as Phase 3 `command_id`.)

### 4.3 New schema: `shared/schemas/command_translations_envelope.json`

The EGS → bridge message. **Cross-file `$ref`s are absolute URIs** (adversarial finding #3 — bare relative refs only resolve by base-URI coincidence and are brittle if any `$id` changes). **`if/then` constraint enforces `valid==false ⇔ command=="unknown_command"`** so a buggy/malicious EGS cannot publish a logically contradictory translation that the bridge would forward to the operator (adversarial finding #2).

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://github.com/ibrahim7860/Gemma-Guardian/shared/schemas/v1/command_translations_envelope.json",
  "title": "EGS → Bridge command translation envelope",
  "type": "object",
  "required": ["kind", "command_id", "structured", "valid", "preview_text", "preview_text_in_operator_language", "egs_published_at_iso_ms", "contract_version"],
  "additionalProperties": false,
  "properties": {
    "kind": {"const": "command_translation"},
    "command_id": {"$ref": "https://github.com/ibrahim7860/Gemma-Guardian/shared/schemas/v1/_common.json#/$defs/command_id"},
    "structured": {"$ref": "https://github.com/ibrahim7860/Gemma-Guardian/shared/schemas/v1/operator_commands.json"},
    "valid": {"type": "boolean"},
    "preview_text": {"type": "string", "minLength": 1, "maxLength": 1024},
    "preview_text_in_operator_language": {"type": "string", "minLength": 1, "maxLength": 1024},
    "egs_published_at_iso_ms": {"$ref": "https://github.com/ibrahim7860/Gemma-Guardian/shared/schemas/v1/_common.json#/$defs/iso_timestamp_utc_ms"},
    "contract_version": {"type": "string", "pattern": "^\\d+\\.\\d+\\.\\d+$"}
  },
  "allOf": [
    {
      "if": {"properties": {"valid": {"const": false}}},
      "then": {"properties": {"structured": {"properties": {"command": {"const": "unknown_command"}}}}}
    },
    {
      "if": {"properties": {"structured": {"properties": {"command": {"const": "unknown_command"}}}}},
      "then": {"properties": {"valid": {"const": false}}}
    }
  ]
}
```

The same absolute-URI convention applies to `operator_commands_envelope.json` (§4.2) — update those `$ref`s to absolute URIs at implementation time.

## 5. Bridge changes (`frontend/ws_bridge/`)

### 5.1 New: subscribe to `egs.command_translations`

Extend `RedisSubscriber` to add a fourth pattern (`egs.command_translations` is exact, not glob, so it joins the fixed-channel list, not the `*` patterns). On message:
1. Validate against `command_translations_envelope`.
2. Drop bridge-only fields (`kind`, `egs_published_at_iso_ms`).
3. Translate `kind: "command_translation"` → `type: "command_translation"`.
4. **Push onto an `asyncio.Queue` (bounded, see below).** Do NOT call `registry.broadcast` synchronously from the subscriber loop.

**Decoupled-broadcaster architecture (adversarial finding #1):** the subscriber's only job is drain Redis fast. Calling `registry.broadcast` inline blocks the subscriber until every WS client either sends or times out. With slow clients + a busy `state_update` channel, the subscriber falls behind, Redis pubsub buffers fill (`client-output-buffer-limit pubsub 32mb 8mb 60`), and Redis disconnects the bridge — triggering a reconnect storm. To avoid this:

- A new `asyncio.Queue[Dict[str, Any]]` with `maxsize=64` lives on `app.state` as `translation_queue`.
- The subscriber's translation handler does `await translation_queue.put(ws_frame)` — fast, never blocks broadcasts.
- A new lifespan task `_translation_broadcaster_loop` drains the queue and calls `registry.broadcast`. Slow clients only delay translation delivery, never `state_update` ingestion.
- Queue overflow (`QueueFull` from a non-blocking `put_nowait`) is handled by dropping the oldest queued translation — operator gets the freshest result, bridge stays alive. Use `try/except QueueFull` with `get_nowait` + `put_nowait` to evict.

**Decision baked in:** translations push as fast as possible but through a queue, not the subscriber's own coroutine. Three lifespan tasks now: emit_loop, subscriber, translation_broadcaster — all cancelled in lifespan teardown before any `await task`.

### 5.2 New: republish inbound `operator_command` to Redis

In the WS receive loop, replace the Phase 3 stub branch (which just acks) with full republish to `egs.operator_commands`:

1. Validate inbound against `websocket_messages.operator_command`.
2. Build envelope: `{kind, command_id, language, raw_text, bridge_received_at_iso_ms, contract_version}`.
3. Defensive validate against `operator_commands_envelope`.
4. `publisher.publish("egs.operator_commands", envelope)`.
5. On success: ack `{type: echo, ack: "operator_command_received", command_id}`.
6. On any failure: error echo using existing `_echo_error` helper.

### 5.3 New: republish inbound `operator_command_dispatch` to Redis

Symmetric to `finding_approval` Phase 3 path. Validate, build operator_actions envelope with `kind: "operator_command_dispatch"`, defensive-validate, publish to `egs.operator_actions`, ack.

### 5.4 New: `finding_id` allowlist guard

In the `finding_approval` branch, after schema validation, check `aggregator.has_finding(finding_id)` before publishing. On miss: `_echo_error(error="unknown_finding_id", finding_id=...)` and skip publish.

`StateAggregator` gains a new method:
```python
def has_finding(self, finding_id: str) -> bool:
    return finding_id in self._findings  # OrderedDict keyed by finding_id
```

**Edge case explicitly accepted:** if a finding ages out of the aggregator's `max_findings` window between operator-side render and click, the approval is rejected. Acceptable because (a) `max_findings=200` in the demo config, well above realistic concurrent finding count, and (b) the failure mode is honest — the dashboard shows "approval not delivered, finding aged out" rather than silently publishing into the void.

### 5.5 Lifespan changes

`subscriber.run()` already handles all psubscribed patterns; we add `egs.command_translations` to the subscription list AND a new `_translation_broadcaster_loop` lifespan task that drains `translation_queue`. Teardown order (per adversarial finding #6 deferred TODO; we keep Phase 3's same-shape teardown but extend it cleanly):

1. Cancel emit_task, subscribe_task, translation_broadcaster_task.
2. Await `subscriber.stop()`.
3. Await all three tasks (swallow CancelledError).
4. `await app.state.publisher.close()`.

`publisher.close()` already runs in lifespan teardown — unchanged.

## 6. Flutter changes (`frontend/flutter_dashboard/`)

### 6.1 `MissionState` extensions

New per-command state machine, parallel to `_findingActions`:

```dart
enum CommandState { sending, translating, ready, dispatched, failed }

final Map<String, CommandState> _commandActions = {};
final Map<String, Map<String, dynamic>> _commandTranslations = {};
String? _activeCommandId; // foreground command in CommandPanel
```

Lifecycle:

| Trigger | Transition |
|---|---|
| Operator clicks DISPATCH on raw text | Generate command_id, `_activeCommandId=cid`, `_commandActions[cid]=sending`, send `operator_command` |
| Bridge ack `operator_command_received` | `sending → translating` |
| `command_translation` arrives | `translating → ready`, store structured + preview |
| WS drops while in `sending`/`translating` | `→ failed` (snackbar: "Translation lost — retry") |
| Operator clicks DISPATCH on preview | Send `operator_command_dispatch`, `ready → dispatched` (optimistic; bridge ack stays `dispatched`) |
| Operator clicks REPHRASE | Clear `_activeCommandId`, drop bookkeeping, return to empty input |
| Translation timeout (15s, per docs/07) | `translating → failed` (snackbar) |

Translation timeout is a `Timer(Duration(seconds: 15), ...)` armed when `_commandActions[cid]=translating`, cancelled on ready/failed/dispatched.

**Decision baked in:** `_activeCommandId` is single-slot — only one command in flight at a time. Operator can't have two translations racing in the panel.

**Orphan rule (adversarial finding #4):** when a fresh `submitOperatorCommand` is called while a prior cid is still in `sending`/`translating`, the prior cid is *immediately orphaned*: its Timer is cancelled, its `_commandActions[cid]` is dropped from the map (not flipped to `failed` — that would fire a misleading snackbar for a command the operator already moved on from), and its entry in `_commandTranslations` is removed. Late ack/translation frames for an orphaned cid pass through `handleEcho`/`applyTranslation` and find no entry; they are logged at debug level and dropped. This bounds memory growth under aggressive REPHRASE/resubmit cycles.

**Dispatch is non-optimistic (adversarial finding #5):** `dispatchActiveCommand` does NOT optimistically transition `ready → dispatched`. Instead it transitions to a new intermediate state `dispatching` and shows a small spinner on the DISPATCH button. Only on receipt of `{ack: "operator_command_dispatch"}` from the bridge does the state advance to `dispatched`. On `redis_publish_failed` for the dispatch frame, state returns to `ready` (not `failed`) with a snackbar "Dispatch send failed — retry" — the translation is still valid, the operator just needs to re-tap. Without this, a transient Redis blip causes a "Dispatched ✓" UI to flip to `failed`, the operator retaps, and the EGS sees a duplicate dispatch on `egs.operator_actions`.

Updated state machine:

```
(absent) → sending → translating → ready → dispatching → dispatched
                                                       └→ ready (on redis_publish_failed echo)
                  → failed (timeout, WS drop, bridge schema reject)
                  └→ (orphaned & dropped on second submit)
```

### 6.1.1 Late-arrival rule (1B)

If a `command_translation` arrives for a command_id whose state is already terminal (`failed`, `dispatched`), `MissionState.applyTranslation` logs at debug level and drops the frame without updating UI. This prevents a 15s-timeout snackbar from being contradicted by a panel that flips back to `ready` at t=16s.

### 6.2 `CommandPanel` rewrite

Five visual states keyed off `_commandActions[_activeCommandId]`:

| State | UI |
|---|---|
| (no active) | Text input enabled, Translate button enabled, no preview pane |
| `sending` | Input disabled, Translate button shows spinner, no preview |
| `translating` | Input disabled, "Translating with Gemma 4 E4B…" subtitle, spinner |
| `ready` (valid=true) | Preview pane shows `preview_text` + `preview_text_in_operator_language`, DISPATCH enabled, REPHRASE enabled |
| `ready` (valid=false → unknown_command) | Preview pane shows the model's `suggestion` in operator's language, DISPATCH disabled with tooltip "Command not understood — rephrase", REPHRASE enabled |
| `dispatching` | Preview pane unchanged, DISPATCH button shows spinner + label "Dispatching…", REPHRASE disabled (operator can't bail mid-publish) |
| `dispatched` | Preview pane shows checkmark + "Dispatched", DISPATCH hidden, NEW COMMAND replaces REPHRASE |
| `failed` | Error banner with retry, REPHRASE re-enabled |

The Translate button replaces today's stubbed DISPATCH button and is enabled when the input is non-empty + connected + no command in flight.

**Language dropdown semantics (2A):** the dropdown is the **operator's preferred response language** — what the EGS uses to populate `preview_text_in_operator_language`. It is NOT a hint about the input language; Gemma 4 E4B auto-detects the input language from `raw_text`. The dropdown label reads "Reply in:" not "Language:" to make this unambiguous in the UI.

**Input retention (1A):** the text input is NOT cleared on Translate. Raw text persists in the input until the operator either (a) sees a `ready` translation and clicks REPHRASE/DISPATCH, or (b) sees a `failed` state and chooses to retry. A bridge-side Redis publish failure leaves the input populated so the operator can re-submit without retyping. The input clears on terminal `dispatched`.

### 6.3 `DroneStatusPanel` validation event ticker

Each drone row gains a small subtitle line:

```
Validation: 2 fails (last: 14:23:11 — DUPLICATE_FINDING)
```

Source: `egsState["recent_validation_events"]` (already arrives, contract 3). Filter to events for this drone; show count + last event's `issue` field + truncated timestamp. Empty if zero.

**Decision baked in:** the ticker is render-only — no animations, no auto-scroll, no expand. The number changing on the next 1Hz tick is the storytelling hook.

### 6.4 Idempotency / forward-compat

`applyStateUpdate` already promotes findings to `confirmed` when upstream sets `approved=true`. Mirror that for commands: if a future `state_update` carries `executed_command_ids: [...]`, promote `dispatched → confirmed`. Spec hook only — no UI for it in Phase 4 (Person 3's seat).

## 7. Stub EGS — `scripts/dev_command_translator.py`

Mirrors Phase 3's `dev_actions_logger.py` pattern: standalone script for local round-trip testing without Person 3's real EGS.

Behavior:
- `redis-py` async client.
- Subscribe to `egs.operator_commands`.
- For each message, validate against `operator_commands_envelope`.
- Hard-coded translation table (no Gemma 4 — this is plumbing, not the demo). **Adversarial finding #9:** drop the bare `"concentr"` substring (false-positives on `concentric`, `concentration`); use only full-word `concentrate` and `concéntrate` matches; normalize accents via `unicodedata.normalize("NFKD", text)` before comparing so `concéntrate` matches whether or not the operator types the accent.
  | `raw_text` (lowercased + accent-folded) | structured |
  |---|---|
  | "recall" / "regresa" / "vuelve" + drone | `{command: "recall_drone", args: {drone_id: detected, reason: "operator request"}}` |
  | "restrict" / "focus" / "concentrate" / "concentrate"-folded-from-`concéntrate` + zone | `{command: "restrict_zone", args: {zone_id: detected}}` |
  | "exclude" / "avoid" / "evita" + zone | `{command: "exclude_zone", args: {zone_id: detected}}` |
  | otherwise | `{command: "unknown_command", args: {operator_text, suggestion: "Try 'recall drone1' or 'focus on zone east'"}}` |
- Build `command_translations_envelope`, validate it, publish to `egs.command_translations`.
- Print one log line per message (`[stub-egs] cid=... raw='...' → command=...`).

This is enough for Person 4 to e2e test the full path including DISPATCH. Person 3 replaces it with a real Gemma 4 E4B caller; the Redis contract is unchanged.
